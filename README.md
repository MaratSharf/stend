# MES Production System

**Manufacturing Execution System (MES)** — веб-приложение для управления производственными заказами и отслеживания их прохождения через 10 рабочих станций с поддержкой подстанций, аутентификации, ролевой модели и QR-сканирования.

## 🚀 Возможности

- ✅ **Пакетное создание заказов** — автогенерация уникальных номеров (`ORD-0001`, `ORD-0002`, ...)
- ✅ **10 производственных станций** — от Приёмки до Отгрузки
- ✅ **Подстанции** — основные станции могут иметь дочерние узлы (1.1, 1.2, 3.1, 6.1)
- ✅ **QR-сканирование** — интеграция с отдельным сервисом на станции 6.1 (Контроль)
- ✅ **Аутентификация** — сессийная (Flask-Login) и API-ключи для интеграций
- ✅ **Ролевая модель** — гибкое управление правами через систему permissions
- ✅ **Кастомные роли** — создание пользовательских ролей с произвольными правами
- ✅ **Автоматическое завершение** заказов после прохождения всех станций
- ✅ **Статистика производства** в реальном времени
- ✅ **SVG-карта производства** — визуализация потока заказов
- ✅ **Тёмная/светлая тема** интерфейса
- ✅ **PostgreSQL** — надёжная СУБД с поддержкой транзакций

## 📦 Установка и запуск

### Предварительные требования

- **Python 3.10+**
- **PostgreSQL 12+**
- **pip** (менеджер пакетов Python)

### Шаг 1: Создание виртуальной среды

```bash
cd mes_production
python -m venv venv
```

### Шаг 2: Активация виртуальной среды

**Windows**:
```bash
venv\Scripts\activate
```

**Linux/macOS**:
```bash
source venv/bin/activate
```

### Шаг 3: Установка зависимостей

```bash
pip install -r requirements.txt
```

### Шаг 4: Настройка PostgreSQL

```sql
-- Создание базы данных и пользователя
CREATE DATABASE mes_production;
CREATE USER mes_user WITH PASSWORD 'mes_password';
GRANT ALL PRIVILEGES ON DATABASE mes_production TO mes_user;
```

### Шаг 5: Настройка конфигурации

Откройте `config.yaml` и убедитесь, что настроены параметры PostgreSQL:

```yaml
database:
  engine: postgresql
  host: localhost
  port: 5432
  name: mes_production
  user: mes_user
  password: mes_password
```

### Шаг 6: Запуск приложения

```bash
python run.py
```

Приложение запустится на **http://localhost:5000**

### Шаг 7: Первый вход

- **Логин**: `admin`
- **Пароль**: `admin`

> ⚠️ **Обязательно смените пароль** при первом входе через страницу `/change-password`.

### Остановка

Нажмите **Ctrl+C** в терминале.

## 👥 Роли и права доступа

### Встроенные роли

| Роль | Уровень | Описание |
|------|---------|----------|
| `viewer` | 1 | Просмотр заказов, станций, статистики |
| `operator` | 2 | Операционные действия (запуск, перемещение, завершение) |
| `admin` | 3 | Полное управление (пользователи, роли, все операции) |

### Система прав (permissions)

Права разделены на категории:

| Категория | Права |
|-----------|-------|
| **Заказы** | `order_view`, `order_create`, `order_launch`, `order_move`, `order_complete`, `order_cancel` |
| **Станции** | `station_view`, `station_control` |
| **Производство** | `production_view`, `map_view` |
| **Пользователи** | `user_view`, `manage_users` |
| **Роли** | `role_view`, `manage_roles` |

## 📡 API Endpoints

### Заказы

| Метод | Endpoint | Доступ | Описание |
|-------|----------|--------|----------|
| GET | `/api/orders` | Login/API | Все заказы (`?status=...`) |
| POST | `/api/orders` | Operator/API | Создать заказы |
| POST | `/api/orders/<id>/launch` | Operator/API | Запустить в производство |
| POST | `/api/orders/<id>/move` | Operator/API | Переместить на след. станцию |
| POST | `/api/orders/<id>/complete` | Operator/API | Завершить вручную |
| POST | `/api/orders/<id>/cancel` | Operator/API | Отменить заказ |
| POST | `/api/orders/<id>/complete-sub` | Operator/API | Завершить подстанцию |
| POST | `/api/orders/<id>/scan-result` | Operator/API | Сохранить результат QR-сканирования |

### Станции и статистика

| Метод | Endpoint | Доступ | Описание |
|-------|----------|--------|----------|
| GET | `/api/stations` | Public/API | Статус всех станций |
| GET | `/api/statistics` | Public/API | Статистика производства |
| GET | `/api/sub-stations` | Login | Подстанции группой |

### Управление пользователями и ролями

| Метод | Endpoint | Доступ | Описание |
|-------|----------|--------|----------|
| GET | `/api/users` | Admin | Список пользователей |
| POST | `/api/users` | Admin | Создать пользователя |
| POST | `/api/users/<id>` | Admin | Обновить пользователя |
| DELETE | `/api/users/<id>` | Admin | Удалить пользователя |
| GET | `/api/roles` | Login | Список ролей |
| POST | `/api/roles` | Admin | Создать роль |
| POST | `/api/roles/<role>/permissions` | Admin | Установить права роли |
| DELETE | `/api/roles/<role>` | Admin | Удалить роль |

### Аутентификация API

Запросы принимаются с **сессией** или **API-ключом**:

```bash
  -H "X-API-Key: ваш-ключ" \
  -d '{"batch":"B1","product_code":"P1","color":"Red","quantity":3}'

# Через сессию (cookie отправляется автоматически браузером)
```

## 🔄 Жизненный цикл заказа

```
[Создание] → buffer
    ↓
[Запуск] → production → станция 1.0 (Приёмка)
    ↓
[Подстанции] → 1.1, 1.2 (завершение через complete-sub)
    ↓
[Перемещение] → move → станция 2.0, 3.0, ...
    ↓
[Станция 6.1] → QR-сканирование (опционально)
    ↓
[Последняя станция] → 10.0 (Отгрузка)
    ↓
[Автозавершение] → completed
```

## 🔒 Безопасность

- **Хеширование паролей:** PBKDF2 через `werkzeug.security`
- **CSRF-защита:** для всех POST-запросов сессийных пользователей
- **API-ключи:** для машинных интеграций (bypass CSRF)
- **Ролевая модель:** granular permissions для каждого действия
